# Домашнє завдання №7. OS та Контейнери

**Обраний варіант:** Варіант A — Аналіз життєвого циклу контейнера

### 1. Запуск

Команда запуску контейнера у фоновому режимі (використано простий Python HTTP-сервер):
`docker run -d --name python_server -p 8080:8080 python:3.9-alpine python -m http.server 8080`

_Коментар: Запускаємо легкий образ Alpine з Python, відкриваємо порт 8080 і вказуємо команду для старту вбудованого веб-сервера._

### 2. Процес

Команда для перегляду процесів усередині контейнера:
`docker exec python_server ps aux`

**Результат спостережень (який процес є PID 1):**
Команда виводить список, де чітко видно, що процес `python -m http.server 8080` має ідентифікатор `PID 1`.

**Пояснення (чому саме він):**
Контейнер — це не віртуальна машина, а просто ізольована група процесів в основній ОС. Процес, який ми передаємо контейнеру як основний (команда при запуску), стає кореневим для цієї ізольованої області, тому отримує PID 1. Життя контейнера жорстко прив'язане до цього процесу: щойно PID 1 завершується (успішно чи з помилкою), контейнер зупиняється.

### 3. Завершення

Команда для зупинки контейнера:
`docker stop python_server`

**Який сигнал отримує процес:**
При виконанні команди `docker stop`, Docker-демон відправляє процесу з PID 1 сигнал **SIGTERM** (сигнал номер 15). Це м'яке прохання коректно завершити роботу, закрити з'єднання та зберегти дані.

**Що відбувається, якщо процес його ігнорує:**
Якщо програма "зависла" або ігнорує сигнал SIGTERM, Docker чекає стандартний grace-період (зазвичай 10 секунд). Після цього він посилає сигнал **SIGKILL** (сигнал номер 9), який операційна система виконує безумовно, миттєво "вбиваючи" процес і зупиняючи контейнер.

### 4. Логи

Команда для перегляду логів:
`docker logs python_server`

**Звідки беруться логи:**
У світі контейнерів логування працює інакше, ніж у класичних Linux-системах. Контейнер зазвичай не пише логи у файли (наприклад, `/var/log/...`). Усі логи, які виводить команда `docker logs`, — це просто перехоплені стандартні потоки виводу (`stdout`) та помилок (`stderr`) нашого головного процесу з PID 1. Docker перехоплює все, що програма "друкує" на екран, і зберігає це у JSON-файлах на хост-машині.
